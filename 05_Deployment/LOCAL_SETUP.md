# Local Setup

How to reconstruct the Travel Without Borders site locally from the extracted
UpdraftPlus backup, using LocalWP. This is the reference document; the
step-by-step actions live in [RESTORE_CHECKLIST.md](RESTORE_CHECKLIST.md), and
verification in [POST_RESTORE_CHECKLIST.md](POST_RESTORE_CHECKLIST.md).

> The backup is **read-only** and is the project's permanent recovery point.
> Never edit anything under `00_Backups/`. See
> [Backup Strategy](../01_Documentation/BACKUP_STRATEGY.md).

## Prerequisites

- Windows 11 with **Docker Desktop** and **LocalWP** installed.
- The extracted backup at
  `00_Backups/2026-06-25_Pre-Development/` (themes, Plugins, uploads, Others, and
  the `*-db.gz` SQL dump).
- WP-CLI (bundled with LocalWP — available via *Local → Open site shell*).
- A decompression tool for `.gz` (7-Zip, or `gzip` via Git Bash).

## Target versions (match production)

| Component | Production (from backup header) | Local target |
| --------- | ------------------------------- | ------------ |
| WordPress | 7.0 | 7.0 (or closest LocalWP offers) |
| PHP | 7.4.33 | **7.4.x** (do not use Local's PHP 8 default) |
| Database | MariaDB 10.11 | MySQL 8 / MariaDB (LocalWP default is fine) |
| Web server | LiteSpeed | nginx or Apache (LocalWP) |
| Multisite | No (`multisite=0`) | Single site |
| Table prefix | `Gv9IkG4cZ_` | **`Gv9IkG4cZ_`** (must match) |

## Key facts that shape the workflow

- The backup contains **`wp-content` + DB only** — no WP core, no `wp-config.php`.
  LocalWP supplies the core and generates `wp-config.php`.
- **Custom table prefix `Gv9IkG4cZ_`** — `wp-config.php` must be edited to match,
  or WordPress will not find its tables.
- **URLs are absolute** (`https://travelwithoutborders.co.uk`) and must be
  search-replaced to the local URL using a serialize-safe tool (WP-CLI).
- **Security/caching must be neutralised before first load** (see below).

## Restoration workflow (summary)

1. Create the LocalWP site (PHP 7.4, note the local URL, e.g.
   `http://travelwithoutborders.local`).
2. Neutralise security/caching **before first load** (rename plugin folders,
   delete the caching drop-in — see below).
3. Overlay `wp-content` (themes, plugins, uploads) from the backup.
4. Set `$table_prefix = 'Gv9IkG4cZ_';` in `wp-config.php`.
5. Import the SQL dump.
6. Serialize-safe URL search-replace to the local domain.
7. Flush rewrites, caches; re-save permalinks.
8. Verify against [POST_RESTORE_CHECKLIST.md](POST_RESTORE_CHECKLIST.md).

The exact, ordered steps with commands are in
[RESTORE_CHECKLIST.md](RESTORE_CHECKLIST.md).

## Plugins to neutralise before first load

Confirmed from the backup (`Others/rsssl-managed-htaccess.lock`,
`Others/advanced-cache.php`):

| Plugin / drop-in | Why | Action before first load |
| ---------------- | --- | ------------------------- |
| Really Simple Security (SSL) | Forces HTTPS → redirect loop / lockout on local `http` | Rename folder `really-simple-ssl` → `really-simple-ssl.off` |
| `advanced-cache.php` (drop-in) | Stale cache drop-in can white-screen the site | Delete from `wp-content/` (regenerated by the cache plugin if needed) |
| WP Rocket | Caching/asset optimisation breaks local assets | Rename `wp-rocket` → `wp-rocket.off` |
| Breeze | Second caching layer | Rename `breeze` → `breeze.off` |
| Wordfence | WAF / `.user.ini` rules | Rename `wordfence` → `wordfence.off` |
| Post SMTP | Would attempt to send real email | Rename `post-smtp` → `post-smtp.off` |
| UpdraftPlus | Not needed locally; can trigger jobs | Rename `updraftplus` → `updraftplus.off` |

Do **not** copy these `Others/` items into the site: `maintenance.php`,
`wflogs/`, `wp-rocket-config/`, `breeze-config/`, `upgrade-temp-backup/`,
`mu-plugins.bk` (a backup, not active mu-plugins).

## Rollback

The local site is disposable; the backup is the recovery point.

- **Before importing the DB:** take a LocalWP snapshot of the clean site so you
  can roll back to a working WordPress without re-creating it.
- **If the import goes wrong:** drop the database and re-import, or restore the
  pre-import snapshot. Do not attempt in-place fixes on a half-imported DB.
- **If the site becomes unreachable:** delete the Local site and start again —
  nothing of value is lost because all source is in `00_Backups/` (untouched) and
  `07_Source/`.
- **Never** point a local tool at the live database or live site.

## Troubleshooting

| Symptom | Likely cause | Fix |
| ------- | ------------ | --- |
| "Error establishing a database connection" | Prefix not set | Set `$table_prefix = 'Gv9IkG4cZ_';` |
| "One or more database tables are unavailable" | Same prefix issue / partial import | Re-import; confirm `Gv9IkG4cZ_` tables exist |
| Infinite redirect to `https://travelwithoutborders.co.uk` | Really Simple SSL active / URLs not replaced | Disable RSSSL; run WP-CLI search-replace |
| White screen / blank page | Stale `advanced-cache.php`, or PHP 8 vs 7.4 | Delete drop-in; set PHP 7.4 |
| Homepage loads but no styles/images | URLs not fully replaced; cache | Re-run search-replace `--all-tables`; flush cache |
| 404 on every page except home | Rewrite rules / permalinks | `wp rewrite flush`; re-save permalinks |
| Can't log in / wrong domain on login | siteurl/home still production | Confirm search-replace ran on options table |
| Fatal error in theme/plugin | PHP version mismatch | Switch Local site to PHP 7.4 |

See also [KNOWN_LOCAL_DIFFERENCES.md](KNOWN_LOCAL_DIFFERENCES.md) for behaviour
that is *expected* to differ locally and should not be "fixed".
